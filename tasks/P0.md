## P0.1 Level-session validation (ISSUES #8 + #9) — CRITICAL
**Context:** `SubmitLevelResult` can be called without starting a level, repeatedly,
with no time check; stars are client-trusted (only clamped).
**Steps:** Active-session record at `RequestLevelStart` (restaurant, index, startedClock);
submit requires + consumes the matching session; elapsed-time plausibility check via a
pure `EconomyMath.minPlausibleSeconds(level, factor)` (factor in GameConfig); server
recomputes `stars = EconomyMath.starsFor(coinsEarned, level.goals)` and ignores
`payload.stars` except for mismatch logging; clear sessions on PlayerRemoving.
**Files:** `Services/LevelService.lua`, `Modules/EconomyMath.lua`, `GameConfig.lua`, specs.
**Accept:** submit-without-start rejected; double-submit rejected; instant submit
rejected; star inflation impossible (spec: 3-star claim with 1-star coins → 1 star);
legit flow still rewards. All existing specs still green.

## P0.2 Remote rate limiting (ISSUES #10)
**Steps:** Per-player token-bucket guard module wrapping every `OnServerInvoke`;
limits table `GameConfig.REMOTE_RATE_LIMITS` (per-remote burst + refill); log + drop
on breach. Pure-test the bucket math.
**Accept:** spam test drops calls without affecting other players; no remote unguarded.

## P0.3 Profile session locking (ISSUES #11)
**Steps:** Lock token (`sessionLock = {jobId, expiresAt}`) acquired in `load()` via
`UpdateAsync` compare; refuse load (kick with retry message) if foreign unexpired lock;
steal if expired; `save()` transform verifies ownership; release on leave/BindToClose.
Pure-test acquire/steal/expire as a state machine.
**Accept:** simulated two-server race (mock store) never loses the newer write;
normal single-server flow unchanged.

## P0.4 Dev/prod DataStore namespacing (ISSUES #12)
**Steps:** `GameConfig.PLACE_ENV` placeId→env map; store name prefixed by env; boot
log states environment; pure-test the mapping. Unknown placeId defaults to "dev".
**Accept:** dev boots write only `dev_`-prefixed store; mapping spec green.

## P0.5 Real-config test suite (ISSUES #14)
**Context:** All 63 specs use mocks; shipped FastFood/Sushi configs are never tested.
Supersedes the "golden levels" idea — those doc tables are design references only.
**Steps:** (a) Invariant suite over every restaurant × all level indices: orders ⊆ menu,
every recipe producible by the restaurant's stations (walk steps vs station configs),
goals strictly ordered, spawn times sorted, customer types valid. (b) Snapshot suite:
pin generated levels {1,10,25,40} per restaurant; serialize deterministically; commit;
assert. (c) Difficulty monotonicity checks across indices.
**Accept:** suite fails if a config typo (bad recipe id, unproducible item) or curve
regression is introduced; current configs pass.
