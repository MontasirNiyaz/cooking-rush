## P1.1 Hub world + RestaurantPlot system
**Goal:** A single streamed food-court plaza map. Each restaurant = a physical building.
A player's owned restaurants visually reflect THEIR state (signage tier, decor tier,
later prestige flair) — and these states are visible to other players on the server.
**Steps:**
1. New `HubService` (server): owns plot assignment-free shared hub; per-player visual
   state is broadcast (each building renders per-viewer? NO — keep it simple: buildings
   show the LOCAL player's state on their client via `HubController`; other players'
   states surface on their avatar/nameplate badge instead. Log this design choice in
   ROADMAP Decisions with rationale: per-viewer building rendering is cheap client-side
   since visuals are local-only).
2. `RestaurantPlot` model schema: `{ restaurantId, doorCFrame, signageVariants = {tier→asset},
   decorVariants = {tier→asset} }` in a new `Shared/Config/HubPlots.lua`. No per-restaurant
   world scripts — `HubController` binds plots by config id.
3. Door interaction opens the level board (existing level-select UI, reskinned as a
   diegetic board at the door). Remove the standalone world-map screen.
4. Visual tiers derive from owned upgrades count (simple mapping in config).
**Files:** `HubService.lua`, `HubController.lua`, `Shared/Config/HubPlots.lua`, map asset,
schema validator extension.
**Accept:** spawn into hub; walk to FastFood door; start level 1; complete; return to hub;
building visual changes after buying 3+ upgrades. World-map screen deleted.

## P1.2 Instant-start onboarding
**Goal:** New player is cooking within ≤15s of first spawn. No modal tutorial walls.
**Steps:**
1. First-session detection (profile flag) spawns the player INSIDE the FastFood kitchen
   with level 1 auto-starting and one customer approaching.
2. New `TutorialController` driven by `Shared/Config/TutorialSteps.lua`:
   `{ id, trigger, highlightTarget, promptText, completeOn }`. Contextual highlight +
   one-line prompt; interruptible; never blocks input.
3. Levels 1–3 keep their authored content; remove any gating that forces sequence
   completion before free play. Teaching steps fire contextually instead.
**Accept:** stopwatch test from fresh profile: first grill interaction ≤15s; tutorial
steps all data-driven; skipping/ignoring prompts never soft-locks.

## P1.3 Interaction + camera model
**Goal:** One generic, phone-first interaction system for all stations.
**Steps:**
1. Implement BOTH ProximityPrompt and direct tap/click on station hitboxes behind a
   `GameConfig.interactionMode` flag; playtest decides; loser is deleted. Record the
   decision in ROADMAP.
2. Oversized invisible hitboxes on mobile (min ~44pt-equivalent).
3. Fixed readable kitchen camera during levels (isometric-ish, configured per restaurant
   in its config: `cameraOffset`, `cameraFov`); free movement only in hub.
4. Portrait one-hand layout for in-level HUD.
**Accept:** every station archetype usable via the chosen mode on a phone viewport;
camera values come from restaurant config; no per-station input code.

## P1.4 Game-feel pass
**Goal:** Centralized juice so feel is tunable as data.
**Steps:** New `JuiceService` (client) consuming `Shared/Config/Effects.lua`:
event id → `{ sound, vfx, uiPop, cameraShake?, pitchRamp? }` for at minimum:
serve_fast / serve_ok / combo_up (escalating pitch) / combo_break / burn / coin_gain /
star_1 / star_2 / star_3 / level_win / level_fail. Pooled emitters + sounds.
**Accept:** all listed events have effects; muting/replacing any effect = config edit
only; zero Instance.new per-event at runtime (pools warm at level start).
